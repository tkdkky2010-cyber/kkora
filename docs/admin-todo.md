## 1. 정산 모니터링 대시보드

매일 7시 KST에 자동 실행되는 `settleDaily` Cloud Function의 결과를 한눈에 확인.

### 화면 구성
- **오늘 정산 요약**
  - 총 참여자 / 성공자 / 실패자 수
  - 풀 총액 / 분배 완료액 / 미분배액
  - 평균 실패율 (사행성 모니터링용)
- **실행 상태**
  - `settleDaily` 함수 실행 시각, 소요 시간
  - 성공 / 부분 실패 / 전체 실패 표시
  - 실패 사유 (Cloud Functions 로그 링크)
- **카카오페이 송금 현황**
  - 송금 시도 N건 → 성공 N건 / 실패 N건 / 응답 대기 N건
  - 응답 대기 5분 이상 → 노란색 경고
  - 응답 대기 10분 이상 → 빨간색 경고

### 데이터 소스
- `challenges/{challengeId}` (Firestore)
- `dailyPool/{date}` (Firestore)
- `transactions/{txId}` (Firestore, type='reward')
- 카카오페이 API 응답 로그 (`functions/logs` 컬렉션 별도 생성)

### 우선순위
**Critical** — 출시 전 필수. 운영 가시성 없이는 사고 대응 불가.

## 2. 부분 실패 재시도

카카오페이 송금 API가 일부 사용자에게만 실패한 경우, 실패 건만 골라서 재시도.

### 발생 시나리오
- 네트워크 일시 오류 (5xx 응답)
- 카카오페이 API rate limit 초과
- 특정 사용자 계정 일시 정지 (drama)
- 카카오페이머니 잔액 부족 (가맹점 측, 거의 없음)

### 화면 구성
- **실패 건 리스트**
  - 사용자 ID / 챌린지 ID / 분배 예정 금액
  - 실패 사유 (카카오페이 응답 코드 + 메시지)
  - 재시도 횟수
  - 마지막 시도 시각
- **재시도 액션**
  - 개별 재시도 버튼
  - 전체 재시도 버튼 (조심스럽게 — 확인 모달 필수)

### 자동 재시도 정책
- **3회까지 자동 재시도** (1분 / 5분 / 30분 간격)
- 3회 모두 실패 시 → 관리자 알림 (FCM / Slack Webhook / 이메일)
- 24시간 이내 수동 처리 안 되면 → 두 번째 알림 + Critical 등급 인시던트

### 우선순위
**Critical** — 출시 전 필수. 분배 실패는 사용자 신뢰의 핵심.

## 3. 사용자 분쟁 조회 + 응답

"내가 분명 성공했는데 보상 못 받았어요" 같은 클레임에 즉시 응답할 수 있는 도구.

### 입력
- 사용자 ID 또는 카카오 ID 또는 카카오페이 CI

### 출력 (한 화면)
- **사용자 기본 정보**
  - 가입일, 레벨, 누적 챌린지 수, 누적 수익
  - 잔액 (DB 기록 vs 카카오페이 sub-account, 두 값 일치 확인)
- **최근 30일 챌린지 이력**
  - 일자별 참여 / 성공 / 실패 / 분배 금액
  - 각 챌린지의 KakaoPay 송금 응답 코드
- **분쟁 가능성 자동 탐지**
  - 성공인데 송금 응답 없음 → 빨간색
  - 송금 응답 있는데 사용자가 못 받음 → 카카오페이 측 문제 (응답 로그 캡처)
  - 잔액 불일치 → 즉시 알림

### 액션
- **수동 재송금** (실제 미송금 확인 시)
- **분쟁 종결** (응답 정상 확인 시 → 사용자에게 카카오페이 응답 로그 공유)
- **카카오페이 CS 에스컬레이션** (카카오페이 측 오류 확인 시)

### 우선순위
**Critical** — 출시 전 필수. 분쟁 응답 24시간 이내 못 하면 환불 분쟁으로 비화.

## 4. 카카오페이 콜백 모니터링

카카오페이가 비동기로 보내는 결과 콜백(웹훅)이 누락되거나 지연되는 케이스 감지.

### 모니터링 대상
- **충전 콜백**: 결제 완료 후 카카오페이 → 우리 webhook
- **송금 콜백**: 분배 송금 완료 후 카카오페이 → 우리 webhook
- **환불 콜백**: 환불 처리 후 카카오페이 → 우리 webhook

### 검출 로직
- 우리가 송금 API 호출한 시각 + 예상 응답 시간(보통 5초~1분)
- 그 시간 + 5분 지나도 콜백 없으면 → 알림
- 그 시간 + 30분 지나도 없으면 → Critical

### 화면 구성
- 미응답 트랜잭션 리스트
- 우리 시스템 → 카카오페이 호출 시각
- 마지막 status 조회 결과
- "수동 status 조회" 버튼 (카카오페이 조회 API 호출)

### 우선순위
**Major** — 출시 직후 1주일 내 구축. 초기엔 수동 모니터링으로 가능.

## 5. 비상 정지 (Kill Switch)

시스템 이상 / 보안 사고 / 법적 명령 발생 시 즉시 챌린지 시작 차단.

### 차단 옵션
- **신규 챌린지 시작 차단** (밤 10시 전 활성화 시 그날 챌린지 0건)
- **충전 차단** (사용자 충전 불가)
- **출금 차단** (사용자 출금 불가)
- **로그인 차단** (계정 보안 사고 시)
- **전체 정지** (위 4개 동시)

### 진행 중 챌린지 처리
- 비상 정지 시점에 이미 진행 중인 챌린지는:
  - **자동 환불 모드**: 시작 시 차감된 예치금 100% 카카오페이 환불 API로 복구
  - **유예 모드**: 챌린지는 마저 진행, 다음 날 정산은 정상
  - 관리자가 모드 선택

### 사용 시나리오
- 카카오페이 API 다운 (정산 불가능)
- DDoS 공격
- 금융감독원 / 사이버수사대 등 외부 명령
- Critical 버그 발견 (예: 잔액 계산 오류)

### 발동 권한
- **2단계 인증** 필수 (관리자 비밀번호 + OTP)
- 발동 로그 자동 기록 (감사 추적용)
- Slack 채널에 즉시 통보

### 우선순위
**Critical** — 출시 전 필수. 보안/법무 대응의 마지막 보루.

## 7. 구현 스택 권장

### 관리자 페이지
- **Next.js 또는 React + Vite** (SPA)
- Firebase Hosting 또는 Vercel 무료 플랜
- Firestore 직접 연동 (서비스 계정으로 read-only 또는 admin 권한)
- 인증: Firebase Auth + 관리자 화이트리스트 (이메일 매칭)

### 백엔드 (관리 액션)
- Cloud Functions 추가 함수:
  - `adminRetrySettlement(challengeId, userId)` — 개별 재시도
  - `adminBulkRetrySettlement(date)` — 일괄 재시도
  - `adminManualRefund(userId, amount, reason)` — 수동 환불
  - `adminKillSwitch(mode, reason)` — 비상 정지
- 모든 함수는 관리자 화이트리스트 검증 후 실행
- 모든 액션은 `adminLogs/{logId}` 에 자동 기록

### 인증 / 보안
- 관리자 이메일은 `.env` 환경 변수 (`ADMIN_EMAILS=a@x.com,b@y.com`)
- Critical 액션 (수동 환불 / kill switch)은 **OTP 추가 인증** 필수
- 모든 관리자 액션 로그는 90일 보관
